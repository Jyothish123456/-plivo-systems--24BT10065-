Flaky Network — Architecture Summary

Dual-Layer FEC Real-Time Media Transport over UDP Relay

1. Problem Statement

The system consists of two independent processes — a sender and a receiver — that communicate exclusively through a hostile UDP relay. The relay randomly drops packets (2–5%), introduces variable delay (10–80ms), reorders, and duplicates. The harness source delivers 160-byte frames to the sender at a strict 20ms interval. The receiver must deliver each frame to the harness player before its playout deadline (t0 + delay_ms + i × 20ms) with less than 1% deadline misses and bandwidth overhead under 2.0×.

2. System Architecture

Harness Source       SENDER              Hostile Relay         RECEIVER            Harness Player

Both sender and receiver are single-threaded C programs using poll() for non-blocking I/O. The sender binds port 47010 (harness input) and sends to port 47001 (relay uplink). The receiver binds port 47002 (relay downlink) and sends to port 47020 (harness player). There is no feedback path — the two processes communicate in one direction only, through the relay.

3. Wire Format

Packet TypeFormatSizePurpose
DATA (0x01) [type:1B][seq:4B BE][payload:160B] 165 B Original frame
FEC K=2 (0x02, count=2) [type:1B][base_seq:4B BE][count:1B][xor:160B] 166 B Pair XOR parity
FEC K=4 (0x02, count=4) [type:1B][base_seq:4B BE][count:1B][xor:160B] 166 B Quad XOR parity

The count field (byte offset 5) distinguishes K=2 pair parity from K=4 quad parity. Both share type tag 0x02. Sequence numbers use big-endian (network byte order) throughout.

4. Sender Design

4.1 Algorithm

  1. Receive a 164-byte frame (4B seq + 160B payload) from the harness on port 47010.
  2. Construct and send a DATA packet (prepend type byte 0x01) to relay port 47001.
  3. XOR the payload into two running accumulators:
    • K=2 accumulator: After every 2 frames, emit a pair FEC packet and reset.
    • K=4 accumulator: After every 4 frames, emit a quad FEC packet and reset.

4.2 Packet Emission Timeline

Frame 0 → DATA₀
Frame 1 → DATA₁ + FEC₂(0,1)
Frame 2 → DATA₂
Frame 3 → DATA₃ + FEC₂(2,3) + FEC₄(0,1,2,3)
              ↑           ↑              ↑
         20ms after F0  20ms after F2  60ms after F0

4.3 Implementation Details

  • Event loop uses poll() with 10ms timeout.
  • FEC is computed as a running XOR — each payload is XORed into the accumulator as it arrives, then the accumulator is zeroed after emission.
  • No feedback socket is bound. The sender does not process NACKs or any receiver communication.
  • The sender runs until t0 + duration + 2s to ensure all frames are processed.

5. Receiver Design

5.1 Recovery Pipeline

LayerTriggerRecovery LatencyCoverage
Direct delivery DATA packet arrives Relay delay only ~94% of frames
K=2 pair FEC FEC₂ or partner data arrives +20ms from partner ~5% of frames
K=4 quad FEC FEC₄ or group data arrives +60ms from group start ~0.5% (when K=2 FEC dropped)

5.2 XOR Reconstruction

Given a group of K frames where exactly 1 payload is missing:

parity   = payload₀ ⊕ payload₁ ⊕ … ⊕ payload_{K-1}
missing  = parity ⊕ (all received payloads)

The receiver identifies the missing frame by sequence number, recovers the 160-byte payload, and reconstructs the 4-byte header from the known sequence number.

5.3 Implementation Details

  • Immediate forwarding: Every DATA packet is forwarded to the player on receipt — no buffering delay.
  • Duplicate detection: A delivered[] boolean array prevents sending the same sequence number twice.
  • Opportunistic recovery: FEC recovery is attempted both when a FEC packet arrives (checking for missing data) and when a data packet arrives (checking if it completes a group with an existing FEC).
  • Static allocation: All buffers are statically allocated global arrays indexed by sequence number — no malloc().
  • The receiver exits after the last playout deadline + 1s slack.

8. Design Choices and Trade-offs

8.1 FEC over Retransmission (ARQ)

Retransmission requires a NACK to traverse the relay to the sender, and the retransmitted packet to traverse it back — two hostile relay transits with a combined round-trip of 40–160ms. At 20ms frame spacing, most retransmissions arrive after the playout deadline. FEC packets are sent proactively and arrive within one relay transit. In iteration 3 of our development, we tested NACK-based retransmission and found it ineffective due to the high round-trip latency.

8.2 Dual-Layer FEC

Single-layer FEC (K=4 only) was tested in iteration 2 and failed on Profile B because: (a) the FEC packet itself has a ~5% chance of being dropped, and (b) multi-loss events within a 4-frame group become significant at 5% loss rate. Adding a faster K=2 layer provides reliable recovery for isolated losses, while K=4 serves as backup when the K=2 FEC packet is itself lost. The combined probability of both FEC layers failing is approximately 0.4% per frame.

8.3 No Feedback Path

In iteration 4, we combined dual-layer FEC with NACK retransmission. While the miss rate was excellent (0.67%), retransmissions pushed bandwidth overhead to 2.07× — exceeding the 2.0× cap. Removing feedback entirely made overhead deterministic at 1.81× with no risk of exceeding the budget.

8.4 Group Size Selection

Larger FEC groups (K=8, K=16) reduce overhead but increase recovery latency. The FEC packet for a K-frame group is sent (K-1)×20ms after the first frame in the group. For K=8, the first frame would need to wait 140ms + relay delay for recovery, exceeding practical playout deadlines. K=2 + K=4 balances low latency (20–60ms recovery) with strong coverage.

9. System-Level Details

9.1 Socket Architecture

  • Sender: 1 receive socket (bind :47010), 1 send socket (to :47001). Single-threaded poll() loop.
  • Receiver: 1 receive socket (bind :47002), 1 send socket (to :47020). Single-threaded poll() loop.
  • All sockets use SOCK_DGRAM (UDP) on 127.0.0.1.

9.2 Memory Model

  • All buffers are statically allocated global arrays — no malloc(), no fragmentation risk.
  • payloads[8000][160] — 1.28 MB for frame storage.
  • fec_parity[groups][160] — ~320 KB for parity data.
  • Boolean flag arrays (received[], delivered[], fec_ok[]) — ~32 KB.
  • Total memory footprint: approximately 1.7 MB.

9.3 Byte Ordering

  • Sequence numbers are stored in big-endian (network byte order) matching the harness format.
  • The sender preserves the harness's big-endian encoding when constructing DATA packets.
  • The receiver uses ntohl() to convert to host order for array indexing, and htonl() when forwarding to the player.

9.4 Timing and Termination

  • Both processes read T0, DURATION_S, and DELAY_MS from environment variables set by the harness.
  • The sender runs until t0 + duration + 2s (safety margin).
  • The receiver runs until the last playout deadline + 1s: t0 + delay_ms/1000 + (n-1) × 0.020 + 1.0.
  • Both use clock_gettime(CLOCK_REALTIME) for sub-millisecond timing precision.
