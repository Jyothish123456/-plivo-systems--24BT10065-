1. System Architecture

The system operates as two independent UDP processes — a sender and a receiver — communicating exclusively through a hostile relay that randomly drops (~2–5%), delays (10–80ms), reorders, and duplicates packets. The harness source feeds 160-byte frames to the sender at a strict 20ms cadence. The sender's job is to encode these frames with enough redundancy that the receiver can reconstruct them despite relay impairments, while staying within a 2.0× bandwidth cap.

2. Wire Format

Three Packet Types

Type Format Size Purpose
0x01 DATA [type:1B][seq:4B BE][payload:160B] 165 B Original frame data
0x02 FEC K=2 [type:1B][base_seq:4B BE][count:1B=2][xor:160B] 166 B Pair parity (fast recovery)
0x02 FEC K=4 [type:1B][base_seq:4B BE][count:1B=4][xor:160B] 166 B Quad parity (backup recovery)

The count field (byte 5) distinguishes K=2 pair FEC from K=4 quad FEC. Both use the same type tag 0x02.

3. Sender Design

Algorithm

  1. Receive a 164-byte frame from the harness source on port 47010.
  2. Forward immediately as a DATA packet (prepend type byte 0x01) to relay port 47001.
  3. Accumulate XOR parity for two concurrent FEC groups:
    • Layer 1 (K=2): XOR every 2 payloads → emit pair FEC packet.
    • Layer 2 (K=4): XOR every 4 payloads → emit quad FEC packet.

Packet Emission Timeline (per 4 frames)

Frame 0 → DATA₀ Frame 1 → DATA₁ + FEC₂(0,1) Frame 2 → DATA₂ Frame 3 → DATA₃ + FEC₂(2,3) + FEC₄(0,1,2,3) ↑ ↑ ↑ 20ms after F0 20ms after F2 60ms after F0

Implementation Details

  • Event loop: poll()-based with 10ms timeout, single-threaded.
  • FEC computation: Running XOR — each payload is XORed into the accumulator as it arrives, then the accumulator is reset after emission.
  • No feedback processing: The sender does not bind port 47004 or handle NACKs. This eliminates unpredictable retransmission overhead.
  • Byte ordering: Sequence numbers are kept in network byte order (big-endian) as received from the harness.

4. Receiver Design

K=4 Quad FEC FEC₄ arrives or group data completes +60ms from group start ~0.5% (K=2 FEC dropped)

FEC Recovery — XOR Reconstruction

Given a group of K frames where exactly 1 payload is missing:

parity =payload₀ ⊕ payload₁ ⊕ … ⊕ payload_{K-1} missing =parity ⊕ (all received payloads)

The receiver knows which frame is missing (by sequence number), so it only needs to recover the 160-byte payload. The 4-byte header is reconstructed from the known sequence number.

Implementation Details

  • Immediate forwarding: Every DATA packet is forwarded to the player the instant it's received — no buffering delay for direct-delivery frames.
  • Duplicate detection: A delivered[] array ensures each sequence number is sent to the player at most once.
  • Opportunistic recovery: FEC recovery is attempted both when a FEC packet arrives (checking for missing data) and when a data packet arrives (checking if it completes a group with an existing FEC).
  • Static allocation: All buffers (payloads[], fec_parity[], flags) are statically allocated arrays indexed by sequence number — no dynamic memory, no allocator overhead.
  • Timing: clock_gettime(CLOCK_REALTIME) for sub-millisecond deadline tracking. Process exits after last playout deadline + 1s slack.

7. Design Choices & Trade-offs

Why FEC over ARQ (Retransmission)?

Automatic Repeat reQuest (ARQ) requires a NACK to traverse the relay to the sender, and the retransmitted packet to traverse it back — two hostile relay transits. With 20–80ms delay per transit, the minimum round-trip is 40–160ms. At 20ms frame spacing, most retransmissions arrive after the playout deadline. FEC packets are sent proactively, arriving within one relay transit of the original data.

Why Dual-Layer FEC?

Single-layer K=4 FEC fails at 5%+ loss because:

  • The FEC packet itself has a ~5% chance of being dropped.
  • Multi-loss events (2+ losses in 4 frames) become significant at 5% loss.

K=2 pairs provide fast, reliable recovery for isolated losses (the common case). K=4 quads act as a safety net for the ~6% of cases where the K=2 FEC packet is dropped. Together, the probability of unrecoverable loss drops to ~0.4% per frame.

Why No Feedback Path?

We tested NACK-based retransmission and found it counter-productive:

  • Retransmissions consumed variable bandwidth, pushing overhead above the 2.0× cap.
  • NACKs and retransmissions are themselves subject to relay drops and delays.
  • By removing feedback entirely, overhead becomes deterministic at 1.81× — no variance, no risk of exceeding the cap.

Why Not Larger FEC Groups?

Larger groups (K=8, K=16) reduce overhead but increase recovery latency: the FEC packet for a K-frame group is sent (K-1)×20ms after the first frame. For the first frame in a K=8 group, recovery requires waiting 140ms + relay delay, which exceeds practical playout deadlines. K=2+K=4 balances low latency (20–60ms recovery) with strong coverage.

8. System-Level Details

Socket Architecture

  • Sender: 1 receive socket (bind :47010), 1 send socket (to :47001). Single-threaded poll() loop.
  • Receiver: 1 receive socket (bind :47002), 1 send socket (to :47020). Single-threaded poll() loop.
  • All sockets are SOCK_DGRAM (UDP) on 127.0.0.1.

Memory Model

  • All buffers are statically allocated global arrays — no malloc(), no fragmentation, no OOM risk.
  • payloads[8000][160] = 1.28 MB for frame storage.
  • fec_parity[groups][160] = ~320 KB for FEC parity data.
  • Boolean flag arrays (received[], delivered[], fec_ok[]) = ~32 KB.
  • Total memory footprint: ~1.7 MB — trivially fits in cache.

Byte Ordering

  • Sequence numbers are always in big-endian (network byte order), matching the harness format.
  • The sender preserves the harness's BE encoding when constructing DATA packets (zero-copy of the seq field).
  • The receiver uses ntohl() to convert to host order for array indexing, then htonl() when forwarding to the player.

Timing & Termination

  • Both processes read T0, DURATION_S, DELAY_MS from environment variables.
  • The sender runs until t0 + duration + 2s (safety margin).
  • The receiver runs until the last playout deadline + 1s: t0 + delay_ms/1000 + (n-1)×0.020 + 1.0.
  • Both use clock_gettime(CLOCK_REALTIME) for sub-millisecond precision.
  • The harness kills both processes when the run ends, so infinite loops are safe.
